## sparse-vs-dense-profile
- issue: https://github.com/PyAutoLabs/autolens_profiling/issues/44
- completed: 2026-06-07
- workspace-pr: https://github.com/PyAutoLabs/autolens_profiling/pull/46 (merged 57b69ea)
- repos: autolens_profiling
- notes: Full instrumentation + measurement comparison of dense (InversionImagingMapping) vs sparse / w-tilde (InversionImagingSparse) imaging inversion paths on A100 80GB at HST scale, plus NSS production-search validation. Headline: sparse uses 7-10x less VRAM per replica on inversion-heavy cells (pix: 931 MB -> 95 MB; del: 922 MB -> 131 MB) at ~10% per-eval slowdown. The chunked-vmap workaround in PyAutoFit#1303 / #1305 is no longer needed for autolens_profiling A100 search runs because un-chunked num_delete=50 vmap now fits comfortably. searches/_setup.py:_build_imaging now calls dataset.apply_sparse_operator() so the inversion factory selects InversionImagingSparse for any model with a Mapper. searches/_samplers.py:build_nss switches to chunk_size=None. NSS validation: MGE sparse 11.0 min/1.60 ms-per-eval (matches dense baseline); pix sparse 320 min/71.95 ms-per-eval (vs Nautilus 46 min / 46.5 ms = NSS 7x slower because of 4.55x more likelihood evals — structural to slice NS, not closeable). Net production guidance: NSS for MGE/parametric cells, Nautilus for pix/delaunay. CPU vs A100 verdict disagreed on per-eval (CPU sparse 41% faster on pix; A100 sparse 10% slower) — XLA fusion and GPU parallelism explanation in memo. Per-step breakdown also disagreed with full-pipeline single-JIT (breakdown README's "XLA fusion caveat" was load-bearing). Framing correction: searches/_setup.py:305's apply_sparse_operator is the interferometer/datacube builder; _build_imaging had no sparse call before this PR, so PyAutoFit#1303 / #1305 NSS A100 OOMs were dense-by-design not silent-fallback. Filename-collision bug found+fixed in resolve_output_paths (was clobbering A100 JSONs across cells when --config-name was set). Matrix-free CG + SLQ for log_det (the original plan-B PyAutoArray prompt) is effectively obsolete — sparse achieves what matrix-free CG would with a simpler implementation and direct Cholesky log_det preserved.
