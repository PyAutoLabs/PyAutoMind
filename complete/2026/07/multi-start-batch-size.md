- summary: Added batch_size to AbstractMultiStartGradient (MultiStartAdam/ADABelief/Lion). The searches vmapped every start's value_and_grad at once, which OOMs memory-heavy likelihoods: measured on an A100 80GB with a pixelized kernel-CDF lens likelihood at n_starts=16 float64, the jvp fusion is f64[16,15361,2,31,512] = 58.13 GiB -> RESOURCE_EXHAUSTED, with no knob to reduce it (only fp32 = science compromise, or fewer starts = defeats multi-start). batch_size=None keeps the single vmap (no regression); when set, tiling is handed to jax.lax.map(batch_size=), which vmaps within a chunk and scans across, handling a ragged final chunk without a second compile. lax.map is used ONLY when batching is requested (with None it degrades to a sequential scan, destroying the default path's parallelism). Numerically inert: identical best_fom/best_params for batch_size in {None,1,4,14,100}; two A100 runs bit-identical. VALIDATED END-TO-END: unbatched OOM'd instantly; batch_size=4 completed a full f64 pixelized multi-start fit on the A100. NOT the same as Nautilus's n_batch (that is Nautilus's own algorithmic knob autofit forwards; this is implementation-level tiling). optax offers nothing here (MultiSteps/apply_every = gradient accumulation over steps for one param set). Full suite 1480 passed. See project_pixelized_gradient_sampler_infeasible.
